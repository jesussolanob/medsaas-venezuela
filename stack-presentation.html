04.3 · Flujo de datos

Cómo entra un dato al sistema

5 flujos principales cubren el 95% de los inputs. Cada uno con su mecanismo de validación, autenticación y persistencia.

📝

1. Formularios públicos

Registro de médico / paciente, booking público.
Usuario llena UI → Server Action valida con Zod → insert en profiles + subscriptions o appointments + patients. No requiere sesión previa.

/register /book/[doctorId]
🔐

2. APIs autenticadas

CRUD del doctor y admin.
Cliente llama fetch('/api/doctor/*') → middleware valida JWT → requireRole(['doctor']) → Supabase client con RLS → insert/update/delete.

50+ endpoints RBAC guards
⚛️

3. RPCs transaccionales

Operaciones críticas con race conditions.
book_with_package(), reschedule_appointment(), change_appointment_status() → SELECT FOR UPDATE bloquea fila → validaciones → update + log atómico.

FOR UPDATE lock
⚡

4. Triggers automáticos

Auditoría e integridad implícitas.
AFTER INSERT/UPDATE en appointments → automático log en appointment_changes_log. AFTER UPDATE en patient_packages.used_sessions → log en package_balance_log.

100% automático
🌱

5. Seeds + Migraciones

Data estática al desplegar.
plan_configs (4 planes), consultation_block_catalog (15 bloques), specialty_default_blocks (44 rows) llegan via sql_migration_v*.sql versionada en git.

25+ migraciones
🌐

6. Scraping externo

Tasas BCV cacheadas.
/api/doctor/exchange-rate → cascada: pydolarve.org → dolarapi.com → bcv.org.ve (HTML scraping). Resultado cacheado 10 min. Usado en consultations.amount_bs.

3 fuentes Cache 10min

10 · Patrones

Decisiones arquitectónicas

Los patrones que sostienen la escalabilidad y la mantenibilidad del sistema.

🏢

Multi-tenant por doctor_id

Cada tabla principal tiene doctor_id. RLS filtra automáticamente: un doctor jamás ve datos de otro.

🎚️

Feature gating

Tabla plan_features controla qué módulos ve cada plan de suscripción.

⚛️

Optimistic locking

FOR UPDATE en RPCs críticas previene race conditions (booking concurrente del mismo slot).

📜

Audit inmutable

Cada cambio en citas y paquetes queda registrado con actor, timestamp, valor anterior y nuevo.

📸

Snapshots inmutables

Las consultas guardan blocks_snapshot al crearse — cambios futuros del doctor no alteran consultas viejas.

🛡️

Defense in depth

Auth → Middleware → Route guards → RLS → RPC checks. 5 capas independientes.

🔄

Source of truth única

La BD es la autoridad. El cliente no calcula estado, solo lee y muestra.

📦

Schema versionado

25+ migraciones SQL en sql_migration_v*.sql — reproducible desde cero.
