Datos Duplicados Detectados
Campos y datos que existen en múltiples tablas, creando riesgo de inconsistencia.
Crítico
appointments vs consultations — Dos tablas para el mismo concepto
El booking crea AMBOS registros: un appointment y una consultation. Campos duplicados entre ambas: chief_complaint, plan_name, payment_method, scheduled_at = consultation_date. El doctor puede actualizar uno sin actualizar el otro, causando divergencia. Además, el FK es bidireccional (appointments.consultation_id y consultations.appointment_id), lo cual es redundante.
Crítico
Datos del paciente desnormalizados en appointments
appointments almacena patient_name, patient_phone, patient_email, patient_cedula directamente en la fila. Estos mismos datos existen en la tabla patients. Si el paciente actualiza su teléfono o email, las citas anteriores quedan con datos obsoletos. Originalmente diseñado para citas de invitados (sin cuenta), pero ahora los invitados también crean registro en patients.
Advertencia
plan_name almacenado en 3 lugares distintos
El nombre del plan/servicio se guarda en: appointments.plan_name, consultations.plan_name, y la fuente de verdad es pricing_plans.name. Si el doctor renombra un plan, las citas históricas quedan con el nombre viejo (esto podría ser intencional para preservar historial, pero no está documentado).
Advertencia
Pagos fragmentados en múltiples tablas
No existe un ledger unificado de pagos. Los datos financieros están en: subscription_payments (pagos de suscripción), consultations.payment_status + consultations.amount (pagos de consulta), billing_documents (facturas), y accounts_payable. Para conocer la posición financiera completa de un doctor se necesita hacer JOIN de 4+ tablas.
Advertencia
Pacientes duplicados potenciales
Un invitado puede agendar cita → se crea registro en patients con source='booking'. Si luego se registra en el portal de pacientes → se crea en profiles con role='patient'. El campo patients.auth_user_id se vincula solo si el email coincide al momento del booking, pero no hay proceso de reconciliación posterior.
Info
FK bidireccional appointments ↔ consultations
appointments.consultation_id apunta a consultations, y consultations.appointment_id apunta a appointments. Solo se necesita una dirección. Esto complica las queries y aumenta la posibilidad de registros desincronizados.
Oportunidades de Mejora
Recomendaciones de arquitectura para escalar a alto tráfico y mantener consistencia de datos.
Arquitectura
1. Unificar appointments + consultations
Crear una sola tabla visits que represente cualquier encuentro médico, con un campo origin (booking | doctor_created) y status (scheduled → confirmed → in_progress → completed → cancelled). Elimina la duplicación de campos y la confusión sobre cuál tabla es la fuente de verdad. La migración puede hacerse gradualmente: mantener ambas tablas con un trigger de sincronización, luego migrar las queries una por una.
Performance
2. Índices compuestos para queries frecuentes
Las queries más comunes son: citas por doctor + fecha, consultas por doctor + mes, pacientes por doctor. Crear índices compuestos: CREATE INDEX idx_visits_doctor_date ON visits(doctor_id, scheduled_at DESC), CREATE INDEX idx_patients_doctor ON patients(doctor_id, created_at DESC). Agregar índice parcial para slots disponibles: WHERE status != 'cancelled'.
Integridad
3. Eliminar datos desnormalizados de appointments
Remover patient_name, patient_phone, patient_email, patient_cedula de la tabla appointments. Siempre hacer JOIN con patients via patient_id. Para citas de invitados sin cuenta, el booking ya crea registro en patients, así que la desnormalización ya no es necesaria.
Finanzas
4. Ledger unificado de transacciones
Crear tabla transactions (id, doctor_id, patient_id, visit_id, type (consultation_payment | subscription | refund), amount, currency, method, status, reference, created_at). Todas las operaciones financieras escriben aquí. Las vistas agregadas (doctor_monthly_revenue, platform_mrr) se construyen sobre esta tabla.
Escalabilidad
5. Particionamiento por fecha para alto tráfico
Para escalar a miles de médicos: particionar visits por scheduled_at (mensual), transactions por created_at. Supabase soporta table partitioning nativo con PostgreSQL. Esto mantiene las queries del mes actual rápidas sin tocar datos históricos.
Automatización
6. Recordatorios automáticos con Edge Functions
Implementar Supabase Edge Function con cron que consulte citas próximas (24h, 2h antes), cruce con appointment_reminders_config del doctor, y envíe via WhatsApp Business API o Resend/SendGrid para email. Registrar cada envío en reminder_logs para trazabilidad. La tabla appointment_reminders_config ya existe — solo falta la función de ejecución.
Seguridad
7. Auditoría de cambios (audit log)
Para compliance médico, crear tabla audit_log (table_name, record_id, action, old_data JSONB, new_data JSONB, user_id, timestamp). Usar trigger genérico en tablas sensibles: consultations, prescriptions, ehr_records. Esto permite reconstruir el historial clínico de cualquier paciente y es requisito para certificaciones de salud digital.
Resiliencia
8. Idempotencia en operaciones críticas
El booking ya usa optimistic lock para paquetes, pero otras operaciones no son idempotentes. Agregar idempotency_key al endpoint de booking y pagos para prevenir doble-submit. Usar transacciones de base de datos (BEGIN...COMMIT) para operaciones multi-tabla como la aprobación de pagos que toca subscription_payments + subscriptions + clinics.
